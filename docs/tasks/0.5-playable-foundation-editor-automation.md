# Task Brief — Playable Foundation Editor Automation

## Title

Automate deterministic Playable Foundation asset setup and evidence capture

## Player or production value

The native third-person foundation builds and tests, but Playable Foundation 0.5 still depends on repetitive Editor asset creation and wiring. A manifest-driven setup and validator reduce the human session to map ownership, spatial judgement, PIE feel and capture evidence.

## Objective

Provide a dry-run-first, idempotent UE 5.8 command-line workflow that can create or repair the five Input Actions, one Input Mapping Context, three derived Blueprints and a collision-free temporary visual, optionally persist the exact sandbox GameMode override behind a separate guard, then validate the complete structural integration.

## Current behaviour

- The native character, PlayerController and GameMode exist at commit `82da258`.
- Seven `TheVeil.*` automation tests pass.
- The required Editor-created input assets and Blueprints have been generated from the checked manifest.
- `BP_TheVeilCharacter` contains one manifest-owned collision-free temporary visual.
- `L_Dev_Sandbox` has the intended GameMode override, one PlayerStart and no placed auto-possessed pawn.
- Strict structural validation passes with no map or content mutation.

## Required behaviour

- One versioned JSON manifest is the source of truth for input assets, exact mappings, Blueprint parents/defaults and structural map expectations.
- Setup defaults to a non-mutating dry run.
- Apply mode requires an explicit switch and refuses to run beside an open Editor.
- Map apply requires the additional `-ApplyMap` guard and can save only `/Game/Maps/L_Dev_Sandbox`'s GameMode override.
- Character visual setup is restricted to one `PlaceholderVisual` Static Mesh component attached to `CapsuleComponent`, with no collision or navigation influence.
- Re-running apply repairs the exact bounded asset set rather than adding duplicate mappings.
- Validation is read-only and emits machine-readable JSON evidence.
- Missing assets are reported as an expected incomplete state without generating misleading commandlet errors when `-AllowIncomplete` is used.
- Build, automation, Editor validation and `git diff --check` can be run through one verification command.

## Acceptance criteria

1. The setup command defaults to dry-run and changes no Content or map assets.
2. Apply requires `-Apply`, refuses an open Editor and is bounded to nine manifest-owned assets; map persistence additionally requires `-ApplyMap` and is bounded to the one declared sandbox.
3. Input mappings match the manifest exactly, including modifier order and controller dead zones.
4. The controller and GameMode Blueprint defaults resolve to the intended assets/classes.
5. Validation checks the sandbox GameMode override, exactly one loaded PlayerStart and zero placed auto-possessed pawns without saving the map.
6. Both tools write JSON reports and retain commandlet logs under ignored `Saved/PlayableFoundation/`.
7. The final documented human session contains only PlayerStart spatial judgement, PIE hardware/feel checks, visual approval and capture evidence.
8. The editor target and all `TheVeil.*` automation tests remain green.

powershell -ExecutionPolicy Bypass -File ".\scripts\Prepare-PlayableFoundationEditor.ps1"
```

powershell -ExecutionPolicy Bypass -File ".\scripts\Validate-PlayableFoundationEditor.ps1" -AllowIncomplete
```

```powershell
powershell -ExecutionPolicy Bypass -File ".\scripts\Verify-PlayableFoundation.ps1" -AllowIncompleteEditorIntegration
```

After the human asset/map session, run the final command without `-AllowIncompleteEditorIntegration`.

To persist the approved integration before final verification:

```powershell
powershell -ExecutionPolicy Bypass -File ".\scripts\Prepare-PlayableFoundationEditor.ps1" -Apply -ApplyMap
```

## Risks

- Unreal Python API names can change between engine releases; the checked capability probe records the tested 5.8 surface.
- Exact manifest repair intentionally removes unlisted mappings from `IMC_Player`; keep unrelated mappings in a different context.
- World Partition may not load a PlayerStart external actor during a headless check; treat a zero count as an Editor loading/evidence issue before moving actors.
- Structural validation cannot prove feel, camera judder, spawn clearance or physical controller behaviour.

## Rollback

Revert the tooling and generated integration commit. Restore only the nine generated Content assets and `L_Dev_Sandbox.umap` through source control. Never delete opaque external actor/object files manually.
