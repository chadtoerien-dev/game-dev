# Task Brief — Playable Foundation 0.5

## Title

Implement Playable Foundation 0.5: third-person character, camera and input

## Player and production value

The native social simulation builds and tests, but the saved sandbox is not yet a controllable action-RPG experience. This milestone creates the playable shell required to test every later social, Tribunal and night-operation system in context.

## Objective

When `/Game/Maps/L_Dev_Sandbox` starts in Play-in-Editor, spawn a controlled third-person character with responsive camera-relative movement, mouse and controller look, sprint, crouch and jump.

## Current behaviour

- Unreal Engine 5.8 builds and opens the project.
- `L_Dev_Sandbox` loads and can enter and exit PIE without a fatal error.
- Five existing `TheVeil.*` automation tests pass.
- The native player character, PlayerController and GameMode are implemented on the feature branch.
- The manifest-owned Enhanced Input assets and derived Blueprints are persisted locally and pass strict structural validation.
- `L_Dev_Sandbox` has the intended GameMode override, one loaded PlayerStart and no placed auto-possessed pawn.

## Required behaviour

- `L_Dev_Sandbox` uses a deliberate The Veil GameMode.
- A The Veil player character spawns at a PlayerStart.
- WASD and left stick provide camera-relative movement.
- Mouse and right stick control the camera.
- The character can sprint, crouch and jump.
- Movement acceleration, braking and rotation are responsive rather than slow or clunky.
- The camera handles collision and does not visibly judder under normal PIE conditions.
- The existing social simulation and automation tests remain unaffected.

## Initial tuning target

These are starting values, not immutable design decisions:

- walk speed: approximately 450–520 cm/s
- sprint speed: approximately 680–760 cm/s
- crouched speed: approximately 250–320 cm/s
- acceleration: approximately 1800–2400 cm/s²
- braking deceleration: approximately 1600–2200 cm/s²
- rotation rate: approximately 500–720 degrees/s
- spring-arm length: approximately 300–380 cm

The implementer must expose or centralise tuning so the human can adjust feel in editor or defaults without rewriting input logic.

## Acceptance criteria

1. The editor target builds successfully with no new compiler errors.
2. All existing five `TheVeil.*` automation tests remain green.
3. `L_Dev_Sandbox` starts with the intended GameMode.
4. Exactly one local player pawn spawns at the intended PlayerStart.
5. Keyboard movement is camera-relative and responsive.
6. Controller movement and look have functional parity.
7. Sprint starts and stops reliably and cannot remain stuck after key release or focus loss.
8. Crouch transitions reliably and respects collision constraints.
9. Jump works only when allowed by normal Character Movement rules.
10. The camera rotates smoothly, follows the character and retracts against geometry.
11. The character does not fall through the landscape or spawn inside geometry.
12. PIE can run for at least two minutes and exit with `Esc` without a crash or fatal error.
13. The implementation does not modify social-state, memory, claim, voting or save behaviour.
14. A short capture demonstrates walking, sprinting, crouching, jumping, camera rotation and collision.
15. Integration documentation lists every editor-created asset and its settings.

Manual PIE test:

1. Open `L_Dev_Sandbox`.
2. Confirm the expected GameMode and default pawn.
3. Press Play in Selected Viewport.
4. Test WASD, mouse look, sprint, crouch and jump.
5. Repeat movement and camera checks with a controller.
6. Walk the character into terrain and test camera obstruction.
7. Alt-tab or change viewport focus while sprint is held, then return and confirm sprint is not stuck.
8. Run for at least two minutes.
9. Press `Esc` and confirm clean PIE shutdown.
10. Capture video and screenshots for Gemini visual review.

## Risks

- Input can be initialised in the wrong lifecycle function or applied repeatedly.
- A Blueprint default pawn may silently override the native GameMode setting.
- World Partition or PlayerStart placement can create apparent movement failures unrelated to C++.
- Editor-generated input assets can be omitted from Git if the commit is not reviewed carefully.
- Over-tuning movement before animation selection can waste work.
- Parallel agents can conflict in character, controller, GameMode and config files.

## Rollback

- Revert the feature branch or PR.
- Restore the previous map GameMode override and PlayerStart state in Unreal Editor.
- Do not delete or regenerate the sandbox World Partition files manually.
- Preserve the verified 0.4 source and tests.
